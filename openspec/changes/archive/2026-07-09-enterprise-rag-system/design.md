## Context

全新构建的企业内部知识库 RAG 系统，无历史遗留代码。目标承载 10k QPS（缓存命中后实际 LLM 压力约 5k QPS），多租户累计 PB 级文档，单租户 TB 级。系统需要在高并发、数据新鲜度、回答准确性三个维度同时达标。

核心约束：
- LLM 私有化部署，GPU 资源有限，必须最大化利用率
- 批量更新（非实时），但更新期间不能停服
- 多租户数据隔离，同时要保持查询效率
- 意图识别延迟必须控制在 15ms 以内，不能成为瓶颈

## Goals / Non-Goals

**Goals:**
- 三类意图（精准问答/比较分析/概念关系）差异化检索路由
- 10k QPS 下 P99 延迟 < 3s
- 语义缓存命中率 ≥ 50%
- 零停机批量索引更新
- 完整的四层评估体系支撑持续迭代

**Non-Goals:**
- 实时文档更新（批量即可）
- 多轮对话上下文超过 5 轮的深度推理
- 跨租户知识共享

## Decisions

### D1: 意图识别采用两阶段而非纯 LLM 分类

**决策**: 规则层（正则 + 关键词）+ MiniLM 小模型分类器，不用 LLM 做意图识别。

**理由**: LLM 分类延迟 100-300ms，在 10k QPS 下意图识别本身就会成为瓶颈。MiniLM 推理 5-10ms，规则层 < 1ms，总延迟 < 15ms。三类意图边界相对清晰，小模型准确率可达 92%+，不需要 LLM 的泛化能力。

**备选**: 纯 LLM 分类（准确率更高但延迟不可接受）、纯规则（无法覆盖边界 case）。

---

### D2: 向量存储按租户分 Collection，非共享 Collection 加过滤

**决策**: Milvus 中每个租户独立 Collection，不共享。

**理由**: 共享 Collection 加 tenant_id 过滤在 PB 级下会导致全量扫描，查询延迟不可控。独立 Collection 使每个租户的 HNSW 索引可以全量驻内存（单租户 TB 级，向量约百 GB 可接受），查询延迟稳定在 10-50ms。代价是 Collection 数量多，但 Milvus 支持数千 Collection。

**备选**: 共享 Collection（运维简单，但查询性能不满足要求）。

---

### D3: 知识图谱选型用 Neo4j，不选分布式图数据库

**决策**: Neo4j 单集群，按租户用 label 隔离，不引入 Nebula Graph。

**理由**: 多租户加起来 PB 但图谱规模远小于文档规模（图谱是文档的提炼）。Neo4j 单集群足以支撑，Cypher 查询成熟，运维复杂度低。Nebula Graph 分布式能力强但运维成本高，当前规模不值得。

**备选**: Nebula Graph（过重）、用向量数据库模拟图（查询能力弱）。

---

### D4: LLM 分级路由——7B 池处理 Intent-1，14B 池处理 Intent-2/3

**决策**: 独立两个模型池，按意图路由，不共享。

**理由**: Intent-1（精准问答）有精确文档支撑，7B 模型完全胜任，延迟低、成本低。Intent-2/3 需要综合推理和关系推断，14B 模型效果明显更好。独立池资源隔离，14B 池繁忙不影响 Intent-1 响应时间。60% 请求走 7B 池，显著降低整体 GPU 成本。

**备选**: 统一 14B 池（运维简单但成本高、延迟高）。

---

### D5: 语义缓存用 embedding 相似度匹配，阈值 cosine > 0.92

**决策**: Redis VSS 存储 query embedding，相似度超过 0.92 视为缓存命中，直接返回历史答案。

**理由**: 企业知识库热点问题高度集中，同一问题有多种问法。精确匹配命中率极低，语义缓存可将命中率提升到 50%。0.92 是在"误命中率"和"漏命中率"之间的经验平衡点，可根据实际数据调整。

**批量更新联动**: 维护 doc_id → cache_key 反向索引，文档更新时精准失效相关缓存，不全量清空。

---

### D6: 离线索引采用 Shadow/Active 双集合原子切换

**决策**: 新索引构建在 Shadow Collection，验证通过后更新 Milvus Collection Alias 切换，原子操作 < 1ms。

**理由**: 避免更新期间查询打到不完整的索引。旧集合保留 24h 作为回滚窗口。三个索引（向量/BM25/图谱）必须全部 READY 才能触发切换，任意失败则整体回滚。

---

### D7: vLLM Prefix KV Cache + 同租户请求亲和路由

**决策**: 同一租户请求通过一致性哈希优先路由到同一 vLLM 实例，最大化 Prefix KV Cache 命中。

**理由**: 企业场景 System Prompt + 租户指令前缀固定（约 300 tokens），同租户请求共享 KV Cache 可节省约 20% prefill 计算。亲和路由实现简单（hash(tenant_id) % N），代价是轻微负载不均衡，用 pending_tokens 感知调度补偿。

## Risks / Trade-offs

**[知识图谱抽取质量不稳定]** → 设置置信度阈值，低置信度关系进入人工审核队列，高置信度自动入图。初期宁可覆盖率低也要保证准确率，避免错误关系污染 Intent-3 结果。

**[语义缓存返回过时答案]** → 文档批量更新后精准失效相关缓存；对时效敏感的 query（含"最新"/"当前版本"等词）强制跳过缓存。

**[7B 模型处理 Intent-1 出现幻觉]** → 检索结果作为强约束，system prompt 明确指示"仅基于提供的文档回答，文档中没有的信息不要推断"。Faithfulness 指标持续监控，低于 0.90 触发告警。

**[vLLM 单实例 OOM]** → 显存利用率监控设 85% 上限告警；PagedAttention 已大幅降低 OOM 概率；发生 OOM 时自动重启实例并从负载均衡中临时摘除。

**[批量更新期间缓存命中率骤降]** → 精准缓存失效（只失效变更文档相关 key）而非全量清空；更新在业务低峰期（凌晨）执行。

**[多租户 Collection 数量膨胀]** → Milvus 支持数千 Collection；设置每租户最小文档量阈值（< 1000 文档的租户共享一个 Collection 加过滤）。

## Migration Plan

1. **基础设施搭建** (Week 1-2): 部署 Milvus、ES、Neo4j、Redis、PostgreSQL、对象存储
2. **离线流水线上线** (Week 3-4): 文档摄入、embedding、图谱抽取，建立初始索引
3. **在线服务上线** (Week 5-6): API Gateway、意图识别、检索层、LLM 层串通
4. **评估体系建立** (Week 7): Golden Set 构建，RAGAs 集成，CI 自动评估
5. **灰度上线** (Week 8): 10% 流量，监控各层指标，调整阈值
6. **全量上线** (Week 9-10): 逐步放量至 100%

**回滚策略**: 每个阶段独立可回滚；Shadow/Active 机制保证索引更新可回滚；LLM 层故障降级到直接返回检索结果。

## Open Questions

- **Q1**: 租户数量上限是多少？影响 Milvus Collection 规划和小租户共享策略的阈值设定。
- **Q2**: 知识图谱的预定义关系类型最终清单需要与业务方确认（目前设计了5类：属于/依赖/替代/集成/概念解释）。
- **Q3**: 多租户隔离是逻辑隔离还是物理隔离（独立 Milvus 实例）？安全合规要求待确认。
- **Q4**: 批量更新频率——目前设计为每日或每周，具体 SLA 需与业务方对齐。
